AV1视频归档压制工具
================================

运行要求
--------
1. Python 3.10 或更高版本（通常自带 tkinter）。
2. 支持 libsvtav1、libopus 和 Matroska 的 FFmpeg，以及配套 ffprobe。

首次运行
--------
首次运行时，如果同名 JSON 不存在，程序会在脚本同目录自动生成：

    AV1压制工具.json

填写 ffmpeg 和 ffprobe 路径后重新运行。可以使用绝对路径，也可以使用相对于脚本目录的路径。
Windows JSON 路径建议使用正斜杠，例如：

{
  "ffmpeg_path": "D:/Tools/ffmpeg/bin/ffmpeg.exe",
  "ffprobe_path": "D:/Tools/ffmpeg/bin/ffprobe.exe"
}

升级与中断恢复
--------------
本版沿用状态文件版本 2，没有提升状态结构版本。
要继续当前版本的未完成任务，请：

1. 先停止并关闭旧脚本；
2. 用本压缩包中的 AV1压制工具.py 覆盖原目录中的同名脚本；
3. 保留原目录中的 AV1压制工具.state.json；
4. 从同一个目录运行脚本，并在提示时选择恢复。

注意：恢复的是任务队列。中途停止的单个视频仍会从头编码；已经完成且验证通过的项目会跳过。
如果把新脚本放到另一个目录运行，它会读取新目录中的状态文件，无法自动看到旧目录的恢复记录。

快速内容指纹
------------
本版会读取源文件开头、约 1/4、1/2、3/4 和结尾附近的数据块，并结合文件大小计算 SHA-256 快速指纹。
大文件通常只额外读取约 1.25 MiB，因此不会像完整文件哈希那样通读整部视频。

新生成的 MKV 会写入来源指纹。之后即使成品改名或在输出目录中移动，只要内容来源一致，工具仍可识别并验证后跳过编码。
快速指纹不是密码学意义上的完整文件校验；最终仍会结合 AV1 10-bit、分辨率、时长、帧率、音轨、字幕、章节和 HDR 等媒体结构验证。

已有记录兼容
------------
当前版本生成但尚未包含快速指纹的状态记录和 MKV 仍可继续使用：
- 旧无收益记录继续按文件大小和纳秒修改时间匹配；
- 旧成品继续按原来源标记、文件名、目录位置和媒体结构验证；
- 新记录和新成品会逐步补充快速指纹。

完整解码验证
------------
重复或非单调 DTS 通常属于时间戳/封装问题，不等于视频画面无法解码。
本版会忽略下列类型的单纯时间戳信息：

    Application provided invalid, non monotonically increasing dts to muxer

只有出现实际解码错误、损坏帧、无效输入数据等其余错误时，才会判定完整解码验证失败。

任务列表
--------
任务列表底部已增加横向滚动条；“状态”列加宽，可横向查看完整失败或跳过原因。

安全提示
--------
- 源文件不会被永久删除。
- 勾选移动原文件时，只有成品验证通过且体积更小，才会移入“_待删除原文件”。
- 无压缩收益任务会写入状态记录，删除无收益成品并保留源文件。
- 删除“_待删除原文件”中的内容前，请自行抽查成品。

------------------------------------------------------------
无人值守稳定性修正版补充
------------------------------------------------------------

本次流程审查和修复：

1. 修复完整解码验证 NameError
   - 删除了误复制进 _full_decode_verify() 的未定义 timeout_seconds、start_time、log_label 引用。
   - 完整解码验证现在使用独立 progress 监控，不再依赖外部变量。

2. 单任务异常隔离
   - 每个视频都有独立异常边界。
   - 某个文件触发 Python 内部异常时，只标记当前任务失败并继续后续队列。
   - 内部异常不会自动移动源文件。
   - 如果编码临时文件已经存在，会保留为 .invalid_internal_时间戳.mkv，避免直接损失数小时编码成果。

3. 真正有效的卡死看门狗
   - 旧代码阻塞读取 FFmpeg stdout；FFmpeg 完全不再输出时，300 秒检测无法执行。
   - 新代码使用独立 stdout 读取线程，主工作线程每 0.5 秒检查一次。
   - 正式编码连续 300 秒媒体时间完全不前进时自动终止并记录失败点。
   - 完整解码验证连续 600 秒无进度时自动终止。
   - 看门狗不限制正常慢速任务的总运行时间。

4. 两阶段队列消除重复整片压制
   - 第一阶段失败后保存失败阶段、时间点和帧号。
   - 第二阶段直接围绕第一阶段失败点做局部测试。
   - 找到可通过方案后才从头完整压制。
   - 不再先重复一次已知会失败的标准整片压制。
   - 修复阶段不会重复执行收益预估。

5. 验证失败定位
   - 完整解码验证持续更新大致媒体位置。
   - 解码验证失败时，第二阶段可以围绕该位置测试。
   - 失败说明新增“大致失败位置”和“最近帧号”。

6. 中断恢复改进
   - state.json 中的“待修复/修复中”任务会恢复为待修复。
   - 恢复后直接排入第二阶段，不重复第一阶段标准压制。
   - 仅在源文件大小、修改时间和快速指纹仍匹配时恢复。

7. 环境错误熔断
   - 检测到输出磁盘空间不足、disk full、no space left 等全局资源错误时停止批次。
   - 避免无人值守时让后续所有文件重复失败。
   - 单文件内容、源文件缺失等问题仍只影响当前任务。

8. 状态文件写入容错
   - 普通 state.json 保存失败会记录错误，但不直接终止当前批次。
   - 无收益记录等涉及文件安全决策的关键写入仍保持原有严格处理。

9. 其他逻辑修复
   - 临时目录校验失败发生在进入“运行中”状态之前，不再导致界面按钮卡死。
   - 设置中修改收益预估阈值后立即同步到当前运行，不再沿用启动时旧值。
   - 主窗口允许横向、纵向拉伸和最大化。
   - 完成摘要改为“完成总数（其中修复成功数）”，避免重复计数造成误解。

说明：
- 任何自动回退都不能数学意义上保证所有损坏或编码器缺陷一定可修复。
- 本版的目标是：单文件问题不拖垮整批任务；已知失败路径不重复浪费；卡死可退出；失败位置和尝试记录可追溯。
- 压缩包不包含 JSON。首次运行会生成新的分组式配置文件。
